## 0. 背景：为什么要做统一框架

原本游戏里**没有可用的聊天**，只有两处遗留实现：

1. `GameRoom` 里一个通用 `chat` 处理器，把消息推到 `state.messages`（上限 50）——但客户端从未接，等于死代码。
2. 公会聊天（`guildChat` 处理器 + 模块级 `onlineGuild` 跨房间广播）——独立实现，与上面那套机制完全不同。

也就是「两套并行、各写各的」的技术债。玩家的问题一针见血：**公会聊天不该是独立特例，而应是一套大聊天框架里的一个频道**。

v1 把两者合并为**单一 `client.send('chat', {channel, text})` 入口 + 服务端单一路由器**，公会仅作为其中一个频道。客户端只有一个 `onMessage('chat')`，最干净。

---

## 1. 频道模型

| 频道 | 含义 | 接收范围 | 发送者 | 路由方式 |
|------|------|----------|--------|----------|
| `world` | 当前地图/房间内公开聊天 | 同房所有客户端 | 任意已入房玩家 | 房间内 `this.clients` 广播 |
| `guild` | 公会频道（原公会聊天） | 同公会**在线成员，跨房间** | 公会成员 | `onlineGuild` 集合投递 |
| `team` | 组队频道 | 同队伍**成员，跨房间** | 队员 | `playerTeam` → 成员 sid → `onlineClients` 投递 |
| `whisper` | 私聊 | 仅目标 + 发送者回显 | 任意（需 `targetCharId`） | `onlineClients` 定向投递 |
| `system` | 系统提示 | 同房所有客户端 | **仅服务端** | 房间内广播（`broadcastChat`） |
| `event` | 全服活动/公告 | **全服在线玩家，跨房间** | **仅服务端**（dev 开关触发） | 全量 `onlineClients` 广播 |

> `event`/`system` 频道**拒绝客户端主动发送**：服务端在 `onMessage('chat')` 里对这两个 channel 直接 return，仅 `broadcastChat()` 方法（服务端调用）可发。防作弊/刷屏。

## 2. 服务端实现（GameRoom.ts）

### 2.1 全量在线注册表

新增模块级 `onlineClients: Map<Client, {charId?: number, name?: string}>`。
- `onJoin`：`registerOnline(client, charId, ch.name)` 登记。
- `onLeave`：`unregisterOnline(client)` 注销。

> 为什么需要它：旧 `broadcastTeam` 用 `room.clients.find(sid)` **只覆盖同房间成员**，跨房间队友收不到。统一聊天用 `onlineClients` 做跨房间投递，guild/team/whisper/event 全部受益。

### 2.2 统一处理器

```ts
this.onMessage('chat', (client, data: { channel?: string; text?: string; targetCharId?: number }) => {
  const channel = data.channel || 'world';
  // 1) 服务端专属频道：客户端不得伪造
  if (channel === 'system' || channel === 'event') return;
  // 2) 取发送者身份
  const charId = sessionCharMap.get(client.sessionId);
  const name = charId !== undefined ? (getCharacter(charId)?.name || '匿名') : '匿名';
  const text = typeof data.text === 'string' ? data.text.trim().slice(0, 200) : '';
  if (!text) return;
  const payload = { channel, fromName: name, fromCharId: charId ?? -1, text, ts: Date.now() };
  // 3) 按频道分流
  if (channel === 'world' || channel === 'system') {
    this.broadcast('chat', payload);                 // 本房间
  } else if (channel === 'guild') {
    if (charId === undefined) return;
    const gid = getMemberGuild(charId);
    if (gid === null) return;
    onlineGuild.get(gid)?.forEach(({ client: c }) => c.send('chat', payload));
  } else if (channel === 'team') {
    const tid = playerTeam.get(client.sessionId);
    if (tid === undefined) return;
    teams.get(tid)?.members.forEach((sid) => {
      const c = onlineClients.get(sid)?.client; if (c) c.send('chat', payload);
    });
  } else if (channel === 'whisper') {
    if (charId === undefined || !data.targetCharId) return;
    onlineClients.forEach((v, c) => { if (v.charId === data.targetCharId) c.send('chat', payload); });
    client.send('chat', payload);                   // 发送者回显
  }
});
```

### 2.3 服务端主动广播（系统/活动）

```ts
private broadcastChat(channel: 'system' | 'event', text: string, target?: Client) {
  const payload = { channel, fromName: '系统', fromCharId: -1, text, ts: Date.now() };
  if (channel === 'system') { this.broadcast('chat', payload); }
  else { onlineClients.forEach((_, c) => c.send('chat', payload)); } // event 全服
}
```

> 现网 `onJoin` 的「X 进入了地图」提示已改走 `broadcastChat('system', ...)`。

## 4. 私密性与安全

- **event/system 服务端专属**：客户端发这两个 channel 被直接丢弃，无法伪造系统广播/全服公告。
- **guild/team 接收范围服务端判定**：客户端只发 `{channel, text}`，能否收到由服务端查 DB/在线表决定（非成员发 guild 自然无人接收）。
- **whisper 定向**：服务端按 `targetCharId` 在 `onlineClients` 查找，仅目标收到 + 发送者回显；非目标收不到（E2E 已验证）。
- 文本统一裁剪 200 字、空文本忽略。

---

## 5. 限制（v1）

- **无持久化**：聊天仅内存环缓冲（guild/team/world 各上限约 50~200，刷新/断连即清空）。历史消息、离线留言留 v2。
- **whisper 按 charId 定向**：客户端从公会/组队成员列表取 `charId` 发起；自由按角色名搜索定向留 v2（需 `name→charId` 查表，暂未实现）。
- **event 由 dev 开关 `/devChat` 触发**：生产环境不设则不暴露该入口，避免误用。
