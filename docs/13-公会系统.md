# 13-公会系统（Guild）

> 版本：v1.0 | 创建：2026-07-16 | 作者：隐世 + 开发代理
> 适用范围：《解》联机版（multiplayer 分支）元系统 Stage D 子项
> 定位：非实时管理走 REST（Express，同端口 2567 `/api`），实时公会聊天走权威游戏服 WebSocket（Colyseus `game` 房间，跨房间广播）
> ⚠️ 2026-07-16 后续：公会聊天已并入**统一聊天系统**（见 [14-统一聊天系统.md](./14-统一聊天系统.md)），现为 `chat` 处理器的 `guild` 频道，原独立 `guildChat` 处理器已废弃。本档管理（REST）部分仍完全有效。

## 一、v1 范围决策（已拍板）

| 项 | 决策 | 说明 |
|---|---|---|
| 功能范围 | **管理 + 公会聊天** | v1 不做等级/经验/仓库/公会技能/贡献等成长系统 |

## 二、数据模型（SQLite，WAL）

三张表，建表幂等（`server/db.ts` 启动 init）。

### `guilds`（公会本体）
| 字段 | 类型 | 说明 |
|---|---|---|
| id | INTEGER PK | 公会 ID |
| name | TEXT UNIQUE | 公会名，2–12 字符 |
| leader_char_id | INTEGER | 会长角色 ID |
| notice | TEXT | 公告，≤200 字符 |
| level | INTEGER | 等级（v1 恒为 1，预留） |
| exp | INTEGER | 经验（v1 恒为 0，预留） |

### `guild_members`（成员）
| 字段 | 类型 | 说明 |
|---|---|---|
| guild_id | INTEGER | 外键 guilds.id |
| char_id | INTEGER UNIQUE | 角色 ID（一人仅属一公会） |
| rank | TEXT | `'leader'` / `'elder'` / `'member'` |

### `guild_applications`（入会申请）
| 字段 | 类型 | 说明 |
|---|---|---|
| id | INTEGER PK | 申请 ID |
| guild_id | INTEGER | 目标公会 |
| char_id | INTEGER | 申请人 |
| message | TEXT | 申请留言，≤200 字符 |
| UNIQUE(guild_id, char_id) | — | 同一公会重复申请幂等拦截 |

## 三、权限模型

三档职位，权重 `rankValue`：`leader=3 > elder=2 > member=1`。

| 操作 | leader | elder | member |
|---|:---:|:---:|:---:|
| 创建公会 | ✓（任意角色） | | |
| 审批入会申请（accept/reject） | ✓ | ✓ | |
| 修改公告 | ✓ | ✓ | |
| 任命/罢免长老（set-rank） | ✓ | | |
| 踢人（kick） | ✓ | ✓ | |
| 转让会长（transfer） | ✓ | | |
| 解散公会（disband） | ✓ | | |
| 退出公会（leave） | （会长须先转让） | ✓ | ✓ |

**权限护栏（`server/guild.ts`）：**
- `kick`：操作者权重须 **严格大于** 目标（`rankValue(my) > rankValue(target)`），即长老不能踢会长/同级，会长不能踢会长自身。
- `set-rank` / `transfer` / `disband`：仅 `leader`。
- `handle-apply` / `set-notice`：`leader` 或 `elder`。
- 会长退出：须先转让会长，否则 `leave` 拒绝。
- 所有操作经 `authChar(req)` 校验 `token` + `charId` 同属一个账号，杜绝跨账号越权操作他人角色。

## 四、REST API 清单（`/api/guild`，均 POST + JSON）

统一返回 `{ ok: boolean, msg?: string, ...data }`。鉴权：`x-token` 头或 body `token` 字段 + body `charId`。

| 路由 | 入参 | 返回要点 | 权限 |
|---|---|---|---|
| `/create` | `name, notice?` | `{ guildId }`；已有所属公会则拒 | 任意 |
| `/info` | `guildId?`（缺省查自己） | 自己：`{ inGuild, myRank, guild{...,memberCount,members[]}, applications?[] }`；他人：`{ inGuild:false }` 或该公会公开信息 | 任意 |
| `/list` | — | `{ guilds: [{id,name,level,memberCount}] }`（取前 50） | 任意 |
| `/apply` | `guildId, message?` | 成功；重复申请/已在公会则拒 | 任意 |
| `/handle-apply` | `applicationId, accept:boolean` | 通过则加成员并删申请；拒绝仅删申请 | leader/elder |
| `/leave` | — | 退出；会长为拒 | 成员 |
| `/kick` | `targetCharId` | 踢出；权重校验 | leader/elder |
| `/set-rank` | `targetCharId, rank('elder'\|'member')` | 改职位 | leader |
| `/transfer` | `targetCharId` | 转让会长并把自己降为 member | leader |
| `/disband` | — | 删公会 + 全成员 + 全申请 | leader |
| `/set-notice` | `notice` | 改公告 | leader/elder |

**约束：** 公会名 2–12 字符、公告/申请留言 ≤200 字符，服务端双重校验。

## 五、实时公会聊天（WebSocket 跨房间广播）

- 客户端 `game` 房间内 `room.send('guildChat', { text })`（文本裁剪前 200 字符，空文本忽略）。
- 服务端 `GameRoom.guildChat` handler：取发送者 `charId → getMemberGuild`；若不在公会则静默丢弃；否则向该公会**在线成员集合（跨房间）**逐一 `client.send('guildChat', payload)`。
- `payload = { fromName, fromCharId, text, ts }`，**含发送者自身**（发送者客户端会收到自己的回声，聊天 UI 据此即时渲染自己发出的消息）。
- 在线成员表 `onlineGuild: Map<guildId, Map<sessionId, {client,charId,name}>>` 为**模块级**（跨房间实例共享）；`onJoin` 时 `registerGuildOnline`，`onLeave` 时 `unregisterGuildOnline`。非成员不在集合中 → 收不到。

> 设计取舍：聊天走游戏服而非 REST 轮询，保证实时与"同公会跨地图房间"语义一致（副本/战斗/地图房间内的同公会成员都能互聊）。

## 六、客户端实现

| 模块 | 文件 | 职责 |
|---|---|---|
| API 封装 | `src/systems/GuildClient.ts` | 镜像 `AuthClient`，`authPost` → `/api/guild/*`；方法：create/info/list/apply/handleApply/leave/kick/setRank/transfer/disband/setNotice |
| 状态字段 | `src/systems/GameStateCore.ts` | `guildId / guildName / guildRank / guildChatLog[]`；`resetCore()` 复位 |
| 面板 UI | `src/ui/panels.ts` → `renderGuildPanel(scene)` | 全屏模态（1000×720）。无公会：创建表单（名/公告）+ 公会列表（申请按钮）；有公会：成员列表（会长可设职/转让/踢人，长老可踢人）、申请审批、公告编辑、退出/解散、公会聊天框（输入框 + 发送） |
| 场景接线 | `src/scenes/GameScene.ts` | `J` 键开关面板（同 `G` 组队键守卫条件）；进房时 `GuildClient.info` 填充 `GameState.guildId/Name/Rank`；`room.onMessage('guildChat', ...)` → `onGuildChat` 推日志并实时追加到聊天区；`sendGuildChat` → `room.send('guildChat', {text})` |
| 输入 | HTML `<input>` 浮层 | 中文名/公告/聊天文本录入，面板销毁时清理 |

**面板按钮权限**：按 `GameState.guildRank` 显隐管理类按钮（设职/转让/踢人/审批/公告/解散）。

## 七、测试（`scripts/test-guild.ts` 端到端烟测）

真机 E2E（非单测）：另起独立临时库的服务端进程（随机空闲端口规避撞端口），用 3 个 colyseus.js 客户端走生产代码路径。覆盖：

**A. REST 管理全链路（20 项断言）**
创建 → 创建者 inGuild/会长/成员数=1 → 未加入者 inGuild=false → 申请 → 会长见 1 条待审 → 审批通过（待审清空、成员数=2、申请人 rank=member）→ 成员退出（inGuild=false、剩 1）→ 重新申请入会 → 设长老 → **长老踢会长被拒** → 转让会长 → 新会长踢旧成员（剩 1）→ 解散（inGuild=false）。

**B. 实时公会聊天**
新公会（bravo 会长 + alfa 成员，charlie 非成员）三客户端进 `game` 房 → alfa 收到 bravo 聊天、charlie 收不到、发送者自身收到回声。

运行：`npx tsx scripts/test-guild.ts`（自动 `tsc -p tsconfig.server.json` 编译服务端，临时库不污染真实 `data.db`）。
**结果：20/20 全绿。**
