## 一、范围决策（已拍板）

| 项 | 决策 | 说明 |
|---|---|---|
| v1 功能范围 | **管理 + 公会聊天** | 已落地 |
| v2 功能范围 | **公会等级 / 经验 / 贡献 / 公会技能 / 行会商店** | 2026-07-16 已端到端落地（本文档各节标注） |
| 加入方式 | **申请 + 会长/长老审批** | 自由加入暂不做 |
| 分工 | **服务端 + 客户端面板一起做** | 端到端可用，非纯接口 |

> 遵循 `12-联机化总体方案.md` 架构分层：公会 CRUD 属"服务后端（非实时元系统）"走 REST；公会聊天属"权威游戏服（实时）"走 WebSocket。

> **v2 设计要点**：公会等级/经验为公会整体成长；贡献分两层——`guilds.contribution`（公会贡献池，由成员活动累积，用于学技能）与 `guild_members.contribution`（个人累计贡献，记录个人贡献度）。公会技能为**全体成员的被动属性加成**，学技能消耗公会贡献池，由会长/长老操作。**行会商店**消费**个人贡献**：此前个人贡献只有"进"没有"出"是死数据，商店让它形成了"任务累积 → 商店兑换专属物资"的个人消费闭环（详见第八节）。所有数值（`expCap=1000*level`、`GUILD_MAX_LEVEL=20`、技能每级 +2%、`costBase=50/step=50`、任务贡献 `gain=max(round(exp*0.5),50)`、商店价格见第八节）均为合理默认，可调。

## 二、数据模型（SQLite，WAL）

三张表，建表幂等（`server/db.ts` 启动 init）。

### `guilds`（公会本体）
| 字段 | 类型 | 说明 |
|---|---|---|
| id | INTEGER PK | 公会 ID |
| name | TEXT UNIQUE | 公会名，2–12 字符 |
| leader_char_id | INTEGER | 会长角色 ID |
| notice | TEXT | 公告，≤200 字符 |
| level | INTEGER | 等级（v2 生效，封顶 `GUILD_MAX_LEVEL=20`） |
| exp | INTEGER | 当前级内经验（v2 生效） |
| contribution | INTEGER | 公会贡献池（v2，成员活动累积，学技能消耗，下限 0） |

### `guild_members`（成员）
| 字段 | 类型 | 说明 |
|---|---|---|
| guild_id | INTEGER | 外键 guilds.id |
| char_id | INTEGER UNIQUE | 角色 ID（一人仅属一公会） |
| rank | TEXT | `'leader'` / `'elder'` / `'member'` |
| contribution | INTEGER | 个人累计贡献（v2，记录个人贡献度，下限 0） |

### `guild_applications`（入会申请）
| 字段 | 类型 | 说明 |
|---|---|---|
| id | INTEGER PK | 申请 ID |
| guild_id | INTEGER | 目标公会 |
| char_id | INTEGER | 申请人 |
| message | TEXT | 申请留言，≤200 字符 |
| UNIQUE(guild_id, char_id) | — | 同一公会重复申请幂等拦截 |

### `guild_skills`（v2 公会技能等级）
| 字段 | 类型 | 说明 |
|---|---|---|
| guild_id | INTEGER | 外键 guilds.id |
| skill_id | TEXT | 技能 ID（`atk`/`def`/`matk`/`mdef`） |
| level | INTEGER | 已学等级（0 未学，封顶 `maxLevel=10`） |
| PK | (guild_id, skill_id) | 唯一，UPSERT 更新 |

> **存量库迁移**：`server/db.ts` 启动时用 `PRAGMA table_info` 检查，若缺 `contribution` 列则 `ALTER TABLE` 幂等补列；`guild_skills` 表按建表模板幂等创建。旧存档不丢数据。

## 三、权限模型

三档职位，权重 `rankValue`：`leader=3 > elder=2 > member=1`。

| 操作 | leader | elder | member |
|---|:---:|:---:|:---:|
| 创建公会 | ✓（任意角色） | | |
| 审批入会申请（accept/reject） | ✓ | ✓ | |
| 修改公告 | ✓ | ✓ | |
| 任命/罢免长老（set-rank） | ✓ | | |
| 踢人（kick） | ✓ | ✓ | |
| 转让会长（transfer） | ✓ | | |
| 解散公会（disband） | ✓ | | |
| 退出公会（leave） | （会长须先转让） | ✓ | ✓ |

**权限护栏（`server/guild.ts`）：**
- `kick`：操作者权重须 **严格大于** 目标（`rankValue(my) > rankValue(target)`），即长老不能踢会长/同级，会长不能踢会长自身。
- `set-rank` / `transfer` / `disband`：仅 `leader`。
- `handle-apply` / `set-notice`：`leader` 或 `elder`。
- 会长退出：须先转让会长，否则 `leave` 拒绝。
- 所有操作经 `authChar(req)` 校验 `token` + `charId` 同属一个账号，杜绝跨账号越权操作他人角色。

## 四、REST API 清单（`/api/guild`，均 POST + JSON）

统一返回 `{ ok: boolean, msg?: string, ...data }`。鉴权：`x-token` 头或 body `token` 字段 + body `charId`。

| 路由 | 入参 | 返回要点 | 权限 |
|---|---|---|---|
| `/create` | `name, notice?` | `{ guildId }`；已有所属公会则拒 | 任意 |
| `/info` | `guildId?`（缺省查自己） | 自己：`{ inGuild, myRank, myContribution, guild{id,name,notice,level,exp,expCap,contribution,skills,memberCount,members[]}, applications?[] }`；他人（指定 guildId）：`{ inGuild:false, guild{...含 level/exp/expCap/contribution/skills/members} }` 公开信息 | 任意 |
| `/list` | — | `{ guilds: [{id,name,level,memberCount}] }`（取前 50） | 任意 |
| `/apply` | `guildId, message?` | 成功；重复申请/已在公会则拒 | 任意 |
| `/handle-apply` | `applicationId, accept:boolean` | 通过则加成员并删申请；拒绝仅删申请 | leader/elder |
| `/leave` | — | 退出；会长为拒 | 成员 |
| `/kick` | `targetCharId` | 踢出；权重校验 | leader/elder |
| `/set-rank` | `targetCharId, rank('elder'\|'member')` | 改职位 | leader |
| `/transfer` | `targetCharId` | 转让会长并把自己降为 member | leader |
| `/disband` | — | 删公会 + 全成员 + 全申请 | leader |
| `/set-notice` | `notice` | 改公告 | leader/elder |
| `/learn-skill` | `skillId`（`atk`/`def`/`matk`/`mdef`） | `{ skillId, level, contribution }`；失败 `msg`：`你不在公会中`/`仅会长长老可学习`/`未知公会技能`/`该技能已达最高等级`/`公会贡献池不足（需 x，现有 y）` | leader/elder |

**`/info` 字段说明（v2 新增）：**
- `guild.exp` / `guild.expCap`：当前级内经验 / 升下一级所需（`expCap = guildExpCap(level) = 1000*level`）。
- `guild.contribution`：公会贡献池总量（学技能消耗来源）。
- `guild.skills`：`{ skillId: level }` 公会已学技能等级。
- `myContribution`：当前角色个人累计贡献（`guild_members.contribution`）。

**`/learn-skill` 流程（v2）：**
1. 鉴权 + 查 `getMemberGuild`；非成员拒。
2. 权限校验：`rank` 须 `leader`/`elder`，否则拒。
3. `getGuildSkillDef(skillId)` 校验技能存在；`cur = getGuildSkillLevel` 达 `maxLevel` 则拒。
4. `cost = guildSkillCost(def, cur) = def.costBase + def.costStep*cur`（默认 `50 + 50*cur`）。
5. 公会贡献池 `< cost` → 拒（提示需 x 现有 y）。
6. `addGuildContribution(gid, -cost)` 扣池 + `setGuildSkillLevel(gid, id, cur+1)` 写等级（UPSERT），返回新等级与剩余贡献池。

**约束：** 公会名 2–12 字符、公告/申请留言 ≤200 字符，服务端双重校验。

> **行会商店购买不是 REST，而是 GameRoom 的 `intent` 消息（`op:'guildBuy'`）。** 原因：学技能扣的是公会级字段（`guilds.contribution`，不在 `PlayerWorld` 里，REST 改 DB 无脏写风险）；而商店发放的是**背包物品 / 称号**（属于 `PlayerWorld.inventory` / `unlockedTitles`），若走 REST 改 DB，玩家在 `GameRoom` 的内存世界是旧副本，离开时会把商店发放覆盖掉。故商店购买复用任务类意图的"内存世界操作 + 统一 `worldSync` 落库"模式，与 `buy`/`mallBuy`/`craft` 等同理。详见第八节。

## 五、实时公会聊天（WebSocket 跨房间广播）

- 客户端 `game` 房间内 `room.send('guildChat', { text })`（文本裁剪前 200 字符，空文本忽略）。
- 服务端 `GameRoom.guildChat` handler：取发送者 `charId → getMemberGuild`；若不在公会则静默丢弃；否则向该公会**在线成员集合（跨房间）**逐一 `client.send('guildChat', payload)`。
- `payload = { fromName, fromCharId, text, ts }`，**含发送者自身**（发送者客户端会收到自己的回声，聊天 UI 据此即时渲染自己发出的消息）。
- 在线成员表 `onlineGuild: Map<guildId, Map<sessionId, {client,charId,name}>>` 为**模块级**（跨房间实例共享）；`onJoin` 时 `registerGuildOnline`，`onLeave` 时 `unregisterGuildOnline`。非成员不在集合中 → 收不到。

> 设计取舍：聊天走游戏服而非 REST 轮询，保证实时与"同公会跨地图房间"语义一致（副本/战斗/地图房间内的同公会成员都能互聊）。

## 八、行会商店（个人贡献消费闭环）

**解决的问题**：v2 落地后个人贡献（`guild_members.contribution`）只有任务领奖时的"进"，没有"出"——是看得到摸不着的死数据。行会商店让它有了消费出口，形成**个人消费闭环**：成员用个人贡献兑换公会专属物资，体现"打工有意义"的正反馈。

### 8.2 购买流程（`guildShopBuy(pw, charId, itemId)`）

1. 校验商品存在（`GUILD_SHOP_ITEMS.find`）；否则拒。
2. 校验角色在公会内（`getMemberGuild` 非 null）；否则拒。
3. 取个人贡献余额（`getMemberContribution`），`< price` → 拒（`个人贡献不足（需 x，现有 y）`）。
4. **称号重复购买护栏**：若 `kind==='title'` 且已在 `pw.unlockedTitles` → 拒（`已拥有该称号`），避免重复扣贡献无收益。
5. 扣个人贡献（`addMemberContribution(charId, -price)`，内部解析公会，下限 0）。
6. 发放：物资类 `world.grantItem(pw, item)`；称号类 `pw.unlockedTitles.push(id)` 且当前未装备时自动 `pw.activeTitle = id`。
7. 返回 `{ ok, msg, data:{ itemId, name, contribution: 剩余个人贡献 } }`。

> 购买成功由 GameRoom 末尾统一 `saveCharacterWorld` 落库，并经 `intentResult`（提示结果）+ `worldSync`（物品/称号到账）下发客户端。面板购买后延迟 400ms 重拉 `/info` 刷新"我的个人贡献"余额显示。

### 8.3 客户端接线

- `src/systems/GuildShop.ts` 镜像目录（仅展示字段），`src/systems/WorldClient.ts` 的 `requestGuildShopBuy(itemId)` → `sendIntent('guildBuy', { itemId })`。
- 公会面板顶部 Tab【公会信息 / 行会商店】；商店 Tab 渲染 **15 张商品卡片**（3 列网格 + 分类标签 HP药/MP药/材料/状态/增益/称号，名/描述/价格内嵌购买按钮，贡献不足置灰"贡献不足"），点击 → `requestGuildShopBuy` → 延迟 `refresh()` 刷新余额。
