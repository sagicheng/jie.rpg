# 灵宠系统（v1.1 重设计方案）

> 状态：**v1.1 已落地**（2026-07-18，提交于 multiplayer 分支；E2E 64/64 全绿；含灵宠蛋来源：战斗概率掉落 + 背包双击开启；含战斗协同：出战灵宠登场参战）
> 范围：**v1.1 = 所有权 + 等级 + 出战 + 光环 + 四大元素 + 五档品质 + 先天技能 + 属性点系统 + 战斗协同（出战灵宠作为第二战斗员参战、宠物技能造成伤害）**
> 设计参照：《梦幻西游》《飘流幻境》——灵宠有技能、有属性点、有元素/品质区分，可多只拥有但仅 1 只协助出战。
> 定位：补「陪伴/养成」维度——此前角色养成只围绕自身属性、装备、鬼道、称号；灵宠提供额外属性光环与后续战斗协同的养成线。
> 前置已就位：账号/DB(SQLite·WAL)、`PlayerWorld` 权威世界 + `worldSync` 广播范式、GameRoom `intent` 意图范式、面板接入范式（`pauseForMenu/resumeFromMenu` + `pointerdown` 守卫）、`recalcStats` 属性链（基础+装备+鬼道+称号+套装）。

---

## 一、核心设计原则（复用现有范式，不造轮子）

1. **灵宠所有权/属性归 `PlayerWorld.pets`，不新建 DB 表。**
   灵宠是 `PlayerWorld` 的一个数组字段（`Pet[]`）。`PlayerWorld` 整体 JSON 序列化后落 `characters.world_data`，新增数组字段**自动持久化**，无需迁移脚本。这符合「反过度工程」偏好——相比公会/好友另开数据表，灵宠天然适合挂在角色世界内。
2. **全部走 GameRoom `intent`，与套装/装备/行会商店同质。**
   发放/出战/收回/放生/分配属性点都改写 `PlayerWorld` 内存状态，必须走权威路径：客户端只发「意图」→ 服务端 `WorldService` 校验改写 → `saveCharacterWorld` 落库 + `worldSync` 广播。杜绝 REST 改 DB 被内存副本覆盖的脏写。
3. **客户端只做展示与光环计算，不持有真相。**
   `worldSync` 到达后 `WorldClient.applyWorldSync` 重建 `GameState.pets`；`recalcStats` 末尾按「出战灵宠」算一层属性光环。战斗流程不改写（v1.1）。
4. **多只拥有、仅 1 只协助出战。**
   `pets[]` 可容纳最多 `PET_SLOT_CAP = 6` 只；`active` 标记出战，服务端强制同一玩家仅一只 `active=true`（切换出战/放生接替均已处理）。

## 二、数据层（`PlayerWorld.pets`，无新表）

```ts
// server/world.ts
export type PetElement = 'fire' | 'wind' | 'water' | 'earth';
export type PetQuality = 'normal' | 'fine' | 'choice' | 'rare' | 'legend';

export interface Pet {
  id: string; speciesId: string; name: string;
  level: number; exp: number;
  element: PetElement;     // 四大元素之一
  quality: PetQuality;     // 五档品质
  hp: number; maxHp: number;
  atk: number; def: number; matk: number; mdef: number; spd: number;
  attrStr: number; attrVit: number; attrAgi: number; attrInt: number; // 已分配属性点
  attrPoints: number;      // 未分配属性点
  skills: string[];        // 先天技能 id（来自物种 skillIds）
  loyalty: number;         // 忠诚度（v1.1 仅记录，暂不影响数值）
  active: boolean;         // 是否出战（同玩家仅一只 active=true）
}

export interface PlayerWorld {
  // ...既有字段...
  pets: Pet[];             // 灵宠栏，上限 PET_SLOT_CAP = 6
}
```

`seedWorld()` 初始化 `pets: []`；`loadFromJSON` 兼容：存量角色无 `pets` / 缺 `element`/`quality`/`skills`/`attr*` 字段时按容错重建（旧档无这些字段不会崩，新字段走默认值）。

| `water` | 水 | 💧 | 0x33ccaa |
| `earth` | 土 | ⛰ | 0xcc8844 |

### 3.2 五档品质 `PET_QUALITIES`
品质影响**成长倍率 `mult`** 与**每级属性点 `attrPerLevel`**，并按权重随机（区域越高传说概率略升）。

| 品质 id | 名称 | 倍率 mult | 每级属性点 | 权重 |
|---|---|---|---|---|
| `normal` | 普通 | 1.00 | 2 | 50 |
| `fine`   | 优秀 | 1.12 | 3 | 28 |
| `choice` | 精良 | 1.25 | 4 | 14 |
| `rare`   | 稀有 | 1.42 | 5 | 6 |
| `legend` | 传说 | 1.65 | 6 | 2 |

`rollPetQuality(zone)`：基础按权重抽；`rare`/`legend` 权重随 `zone` 提升（`boost = min(0.4, (zone-1)*0.03)`，`legend` 加成翻倍），高区域更易出好品质。

## 四、物种定义（`PET_SPECIES`，服务端权威，8 种）

每个物种含 `element` 与 `skillIds`（先天技能），属性快照由 `computePetStats` 按物种 base + growth + 品质倍率 + 已分配属性点计算。

| 物种 id | 名称 | 元素 | 主要定位 | 获取区域 | 先天技能 |
|---|---|---|---|---|---|
| `fox_fire` | 赤焰狐 | 火 | 魔攻/速度 | 2 | 烈焰吐息 |
| `tortoise_rock` | 玄岩龟 | 土 | 血厚/防高 | 3 | 岩石壁垒 |
| `hawk_wind` | 青翼鹰 | 风 | 速度/物攻 | 4 | 疾风之刃 |
| `serpent_water` | 碧水蟒 | 水 | 魔攻/血量 | 5 | 碧水纱 |
| `wolf_shadow` | 暗影狼 | 风 | 物攻/速度 | 6 | 暗影撕咬 |
| `bear_earth` | 撼地熊 | 土 | 血/防/物攻全面 | 7 | 撼地重击 |
| `rabbit_spirit` | 灵兔 | 水 | 魔防/速度 | 8 | 灵兔祝福 |
| `dragonet` | 幼麟 | 火 | 六维均衡·成长极高 | 9 | 祥瑞 |

`computePetStats(sp, level, quality, attrs)`：
```
q = PET_QUALITIES[quality].mult
stat = round( (base + growth*(level-1) + attrContribution) * q )
attrContribution（PET_ATTR_WEIGHT）:
  str → atk +3
  vit → hp +12, def +1
  agi → spd +2
  int → matk +2, mdef +1
```
即：品质倍率整体放大六维；属性点按权重定向加成（力主物攻、体主血防、敏主速、灵主魔攻魔防）。

## 五、服务端方法（`WorldService`，server/world.ts）

| 方法 | 行为 | 边界 |
|---|---|---|
| `createPet(pw, speciesId, name?, zoneOverride?)` | 生成 Lv1 灵宠（**随机品质** + 元素/技能来自物种 + 属性点 0），首只自动出战；`zoneOverride` 覆盖品质随机的区域（开蛋用，按蛋掉落区域定品质） | 物种未知/栏位满(≥6) 拒绝 |
| `openPetEgg(pw, itemId)` | **开启灵宠蛋**：背包找到 `pet_egg` → 随机物种 + 按蛋携带 `zone` 定品质 → `createPet` → 消耗一枚蛋；返回孵化结果 | 无蛋/栏位满(≥6) 拒绝 |
| `grantPetEgg(pw, zone=1)` | Dev/测试用：直接发放一枚 `pet_egg`（带 `zone`）到背包 | — |
| `setActivePet(pw, petId)` | 设出战，遍历把其他 `active=false` | 灵宠不存在拒绝 |
| `addPetExp(pw, petId, amount)` | 加经验+自动升级（重算快照、回满血）；**每升 1 级按品质 `attrPerLevel` 发放未分配属性点** | 灵宠不存在/物种缺失拒绝；上限 Lv70 |
| `setPetAttr(pw, petId, attr, delta)` | 分配/回收属性点：校验 `attr∈{str,vit,agi,int}`；`delta>0` 需 `attrPoints>0`，`delta<0` 需该属性>0；改写后重算属性快照 | 灵宠不存在/物种缺失/未知属性/点数不足 拒绝 |
| `releasePet(pw, petId)` | **驱逐/丢弃**：放生删除；若放生的是出战则首只接替出战 | 灵宠不存在拒绝 |
| `recallPet(pw, petId)` | 收回出战（不删除，可无出战灵宠） | 灵宠不存在拒绝 |
| `getActivePet(pw)` | 取当前出战灵宠（无则 null） | — |
| `grantPetTest(pw, speciesId?, level?)` | Dev 作弊键用：指定或随机物种发放；可选 `level`（灌经验到指定等级，便于测试属性点） | 同 createPet |

升级曲线：`petExpForLevel(level) = 80 * level`（即 Lv1→2 需 80，Lv2→3 需 160，线性）。

## 六、意图接线（`GameRoom.ts` intent switch）

七个 intent，均 `world.*` 改写 + `saveCharacterWorld` 落库 + `worldSync` 广播：

```ts
case 'petSetActive': { ... res = world.setActivePet(pw, data.petId); ... }
case 'petRelease':  { ... res = world.releasePet(pw, data.petId);  ... }
case 'petGrantDev': { ... res = world.grantPetTest(pw, data.speciesId, data.level); ... } // Dev 用，支持 level
case 'petRecall':   { ... res = world.recallPet(pw, data.petId);   ... }
case 'petSetAttr':  { ... res = world.setPetAttr(pw, data.petId, data.attr, Number(data.delta)||0); ... }
case 'usePetEgg':   { ... res = world.openPetEgg(pw, data.itemId); ... }   // 双击灵宠蛋开启
case 'petGrantEgg': { ... res = world.grantPetEgg(pw, data.zone);   ... }  // Dev/测试：发蛋
```

客户端桥接（`src/systems/WorldClient.ts`）：
- `requestPetSetActive(petId)` / `requestPetRelease(petId)` / `requestPetRecall(petId)` / `requestPetGrantDev(speciesId?, level?)` / `requestPetSetAttr(petId, attr, delta)` / `requestUsePetEgg(itemId)`
- `applyWorldSync` 重建 `GameState.pets = pw.pets`（开蛋后自动刷新面板）
- 客户端镜像常量：`PET_ELEMENTS_CLIENT` / `PET_QUALITIES_CLIENT` / `PET_SKILLS_CLIENT` + 辅助 `petElementInfo(el)` / `petQualityInfo(q)` / `petSkillNames(pet)`（仅展示用）

### 6.1 灵宠蛋来源（战斗概率掉落 + 背包双击开启）

**掉落**：`generateLoot(type, zone)`（`src/systems/BattleData.ts`，BattleRoom 与副本共用）按敌人类型加权概率追加 `pet_egg`，并携带掉落 `zone`：

```ts
const eggChance: Record<EnemyType, number> = { 杂妖: 0.015, 恶妖: 0.03, 妖将: 0.06, 妖王: 0.12 };
if (Math.random() < (eggChance[type] ?? 0.08))
  loot.push({ id: 'pet_egg', name: '灵宠蛋', type: 'pet_egg', desc: '双击开启，随机孵化一只灵宠', quantity: 1, zone });
```

蛋作为普通 loot 进入 `pw.inventory`，≤1 秒内经 GameRoom 周期 `worldSync` 进客户端背包（与装备/材料掉落同链路）。Boss/妖将掉率更高，鼓励挑战高强度战斗。

**开启**：背包面板（`renderInventoryPanel`）材料段后新增「灵宠蛋（双击开启）」区，列出 `type==='pet_egg'` 物品；**双击**（350ms 内两次 `pointerdown`）触发 `requestUsePetEgg(item.id)` → 服务端 `openPetEgg`：随机物种（→随机元素/技能）+ 按蛋 `zone` 定品质，`createPet` 生成后消耗一枚蛋；`worldSync` 刷新背包（蛋-1）、`intentResult` 提示孵化结果（含元素·品质）。离线时提示「灵宠蛋需联网开启」（灵宠服务端权威，不复制客户端逻辑）。

**驱逐/丢弃**：即现有「放生」（`releasePet`，出战接替逻辑已处理），玩家可在 U 面板放生多余灵宠。

### 6.2 战斗协同（出战灵宠登场，v1.1 落地）

出战灵宠作为**同一客户端的第二战斗员**进入权威战斗房，拥有独立 SID（`{battleSid}:pet`）、独立回合与独立指令；人物与灵宠在同一指令阶段分别被指挥（按 `actorSid` 分别提交/守卫）。

**服务端（`server/rooms/BattleRoom.ts`）**
- `CombatPlayer` 新增 `isPet:boolean` / `ownerSid:string`（`server/schema.ts`）。
- `onJoin`：若 `options.pet` 携带出战宠 DTO（属性快照 + 技能列表 + 宠MP），生成宠物 `CombatPlayer`（`isPet=true`、`ownerSid=传入的地图房ownerSessionId`）；其技能负载存入 `loadouts[petSid].petSkills`。
- `receiveAction`：按 `data.actorSid`（`client.sessionId` 或 `client.sessionId+':pet'`）分别接收人物/宠物指令；宠物仅允许 `attack`/`defend`/`petSkill`。
- `executePlayerAction` 新增 `petSkill` 分支：按元素用宠物 `atk`(物理)/`matk`(魔法) 结算伤害（系数/伤害类型/MP 取自 `PET_SKILLS`），MP 不足自动转普攻（避免空过）。
- `checkCommandReady` / `onCommandTimeout` / `spdCmp` / `executeEnemyAction` 等按 SID 泛型处理，**宠物天然兼容**（指令阶段需人物+宠物双方都提交才开战；超时宠物自动普攻）。
- `onLeave`：主人离场同步击倒其出战灵宠（宠物非独立连接）。

**客户端（`src/scenes/MultiBattleScene.ts` + `GameScene.buildBattleLoadout`）**
- `buildBattleLoadout` 把 `GameState.pets` 中 `active` 的宠映射为战斗 DTO（`stats`/技能/宠MP=`30+等级*4`），随 `loadout.pet` 下发。
- 宠物卡片在玩家列渲染（紫调描边 + 🐾 前缀），与人物区分；MP 文本额外显示 `🐾MP`。
- 命令栏新增「🐾宠物」按钮（左侧独立）：点开 `*指挥灵宠*` 菜单 → `攻击` / `✨宠物技能`（子菜单列先天技能）/ `🛡防御`；宠物指令经 `send(action, petSid)` 提交，`submittedActors` 按 actor 分别守卫（人物已提交不挡宠物）。
- 指令阶段提示：人物已选、宠物待命时显示「灵宠待命…（点🐾指挥灵宠）」。

> 注：战斗站位已按《梦幻/飘流》式——怪物 2 列×4 行；玩家侧左侧单列，出战灵宠自动排在人物下方（index+1），天然支持「4 人 + 4 灵宠」扩展。

## 七、客户端：光环、面板、养成

### 7.1 属性光环（`GameStateStats.recalcStats` 末尾）
出战灵宠按比例提升玩家最终属性（吃灵宠**当前**属性快照，已含品质倍率与属性点加成）：

```
HP  +10%  (maxHp * 0.10)
ATK +20%  (atk * 0.20)
DEF +20%  (def * 0.20)
MATK+20%  (matk * 0.20)
MDEF+20%  (mdef * 0.20)
SPD +20%  (spd * 0.20)
```
灵宠无 MP 字段 → 光环不含 MP。`computePetAura(pet)`（src/systems/PetSystem.ts）统一计算，E2E 已校验公式对齐。

### 7.2 面板（U 键，`src/ui/panels.ts`）
- 顶部容器 + 遮罩 + `pauseForMenu`，`pointerdown` 守卫冻结移动（与 B/C/J 面板同范式）。
- 每张灵宠卡：图标方块(物种色)、名称、Lv、**元素徽章**(图标+名称+色)、**品质徽章**、出战徽章、EXP 进度条、六维数值、技能名行(来自 `petSkillNames`)。
- **属性点分配行**（力/体/敏/灵 各 `+/-` 按钮，调用 `requestPetSetAttr`）：升级获得未分配点后在此定向加点，实时重算六维。
- 按钮：`出战/收回` 切换（toggle active）、`放生`（release）。
- 底部：出战灵宠光环预览行。
- **C 面板不再含灵宠入口**（按需求移除拍卖行与灵宠入口；灵宠面板由 U 键直接打开）。

### 7.3 Dev 快捷键
`Ctrl+Y`：联机下 `requestPetGrantDev()` 发放随机物种灵宠（落库避免重连丢失）；**离线/单机下本地 `createPetLocal(zone)` 直接造宠**（首只自动出战），与其他 Dev 键（Ctrl+E 等）离线行为一致，便于单机调试灵宠系统。

## 八、端到端烟测（`scripts/test-pet.ts`，E2E 64/64）

与真实联机完全一致：编译服务端 → 临时库起 Colyseus 权威服（随机端口）→ colyseus.js 客户端走生产路径。覆盖：

- **A 发放**：随机物种成功；首只自动出战（`pets.length=1, active=true`）
- **B 发放指定物种**：`fox_fire` 成功；长度=2 且仅一只出战
- **I 新字段**：发放灵宠含 `element∈{火/风/水/土}`、`quality∈{普通/优秀/精良/稀有/传说}`、`skills` 非空且含该物种先天技能（`fox_fire`→`flame_breath`）；属性点字段结构齐全
- **C 切换出战**：`setActive` 后仅目标出战，原出战收回
- **D 收回**：`recall` 后无出战灵宠
- **E 放生**：`release` 后长度递减；若放生的是出战则首只接替
- **J 属性点分配 `petSetAttr`**：
  - 1 级宠 `attrPoints=0` 时正向分配被拒（「无可用属性点」）
  - 发放 5 级宠（必带属性点）后分配 1 点力量：`attrPoints-1`、`attrStr+1`、`atk` 严格增大（str.atk=3 且品质倍率≥1）
  - 反向分配 -1 后属性点与属性复原
  - 非法属性名被拒（「未知属性」）
- **F 栏位上限**：满 6 后第 7 只被拒（含「上限」提示）
- **G 持久化**：`world_data` 落库 pets=6、仅一只是出战；断连新客户端重连恢复=6
- **H 光环公式**：`computePetAura` 的 HP/ATK/DEF/MATK/MDEF/SPD 与 10%/20% 比例逐项对齐
