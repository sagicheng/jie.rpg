# 灵宠系统（v1）技术方案

> 状态：**v1 已落地**（2026-07-18，提交于 multiplayer 分支；E2E 28/28 全绿）
> 范围：**v1 = 所有权 + 等级 + 出战 + 光环**（战斗协同留 v2，见末尾延期项）
> 定位：补「陪伴/养成」维度——此前角色养成只围绕自身属性、装备、鬼道、称号；灵宠提供额外属性光环与后续战斗协同的养成线。
> 前置已就位：账号/DB(SQLite·WAL)、`PlayerWorld` 权威世界 + `worldSync` 广播范式、GameRoom `intent` 意图范式、面板接入范式（`pauseForMenu/resumeFromMenu` + `pointerdown` 守卫）、`recalcStats` 属性链（基础+装备+鬼道+称号+套装）。

---

## 一、核心设计原则（复用现有范式，不造轮子）

1. **灵宠所有权/属性归 `PlayerWorld.pets`，不新建 DB 表。**
   灵宠是 `PlayerWorld` 的一个数组字段（`Pet[]`）。`PlayerWorld` 整体 JSON 序列化后落 `characters.world_data`，新增数组字段**自动持久化**，无需迁移脚本。这符合「反过度工程」偏好——相比公会/好友另开数据表，灵宠天然适合挂在角色世界内。
2. **全部走 GameRoom `intent`，与套装/装备/行会商店同质。**
   发放/出战/收回/放生都改写 `PlayerWorld` 内存状态，必须走权威路径：客户端只发「意图」→ 服务端 `WorldService` 校验改写 → `saveCharacterWorld` 落库 + `worldSync` 广播。杜绝 REST 改 DB 被内存副本覆盖的脏写。
3. **客户端只做展示与光环计算，不持有真相。**
   `worldSync` 到达后 `WorldClient.applyWorldSync` 重建 `GameState.pets`；`recalcStats` 末尾按「出战灵宠」算一层属性光环。战斗流程不改写（v1）。

## 二、数据层（`PlayerWorld.pets`，无新表）

```ts
// server/world.ts
export interface Pet {
  id: string; speciesId: string; name: string;
  level: number; exp: number;
  hp: number; maxHp: number;
  atk: number; def: number; matk: number; mdef: number; spd: number;
  loyalty: number;   // 忠诚度（v1 仅记录，暂不影响数值）
  active: boolean;   // 是否出战（同玩家仅一只 active=true）
}

export interface PlayerWorld {
  // ...既有字段...
  pets: Pet[];       // 灵宠栏，上限 PET_SLOT_CAP = 6
}
```

`seedWorld()` 初始化 `pets: []`；`loadFromJSON` 兼容：`if (Array.isArray(data.pets)) w.pets = data.pets;`（存量角色无 pets 时服务端按 `[]` 容错）。

## 三、物种定义（`PET_SPECIES`，服务端权威，8 种）

| 物种 id | 名称 | 元素 | 主要定位 | 获取区域 | 自带技能 |
|---|---|---|---|---|---|
| `fox_fire` | 赤焰狐 | 火 | 魔攻/速度 | 2 | 烈焰吐息 |
| `tortoise_rock` | 玄岩龟 | 土 | 血厚/防高 | 3 | 岩石壁垒 |
| `hawk_wind` | 青翼鹰 | 风 | 速度/物攻 | 4 | — |
| `serpent_water` | 碧水蟒 | 水 | 魔攻/血量 | 5 | — |
| `wolf_shadow` | 暗影狼 | 暗 | 物攻/速度 | 6 | — |
| `bear_earth` | 撼地熊 | 土 | 血/防/物攻全面 | 7 | — |
| `rabbit_spirit` | 灵兔 | 光 | 魔防/速度 | 8 | — |
| `dragonet` | 幼麟 | 稀有 | 六维均衡·成长极高 | 9 | 祥瑞 |

属性快照按物种 base + growth 计算：`stat(level) = round(base + growth * (level-1))`（见 `computePetStats`）。

## 四、服务端方法（`WorldService`，server/world.ts）

| 方法 | 行为 | 边界 |
|---|---|---|
| `createPet(pw, speciesId, name?)` | 生成 Lv1 灵宠（属性快照），首只自动出战 | 物种未知/栏位满(≥6) 拒绝 |
| `setActivePet(pw, petId)` | 设出战，遍历把其他 `active=false` | 灵宠不存在拒绝 |
| `addPetExp(pw, petId, amount)` | 加经验+自动升级（重算快照、回满血） | 灵宠不存在/物种缺失拒绝；上限 Lv70 |
| `releasePet(pw, petId)` | 放生；若放生的是出战则首只接替出战 | 灵宠不存在拒绝 |
| `recallPet(pw, petId)` | 收回出战（不删除，可无出战灵宠） | 灵宠不存在拒绝 |
| `getActivePet(pw)` | 取当前出战灵宠（无则 null） | — |
| `grantPetTest(pw, speciesId?)` | Dev 作弊键用：指定或随机物种发放 | 同 createPet |

升级曲线：`petExpForLevel(level) = 80 * level`（即 Lv1→2 需 80，Lv2→3 需 160，线性）。

## 五、意图接线（`GameRoom.ts` intent switch）

四个 intent，均 `world.*` 改写 + `saveCharacterWorld` 落库 + `worldSync` 广播：

```ts
case 'petSetActive': { ... res = world.setActivePet(pw, data.petId); ... }
case 'petRelease':  { ... res = world.releasePet(pw, data.petId);  ... }
case 'petGrantDev': { ... res = world.grantPetTest(pw, data.speciesId); ... } // Dev 用
case 'petRecall':   { ... res = world.recallPet(pw, data.petId);   ... }
```

客户端桥接（`src/systems/WorldClient.ts`）：
- `requestPetSetActive(petId)` / `requestPetRelease(petId)` / `requestPetRecall(petId)` / `requestPetGrantDev(speciesId?)`
- `applyWorldSync` 重建 `GameState.pets = pw.pets`

## 六、客户端：光环与面板

### 6.1 属性光环（`GameStateStats.recalcStats` 末尾）
出战灵宠按比例提升玩家最终属性（吃灵宠当前属性快照）：

```
HP  +10%  (maxHp * 0.10)
ATK +20%  (atk * 0.20)
DEF +20%  (def * 0.20)
MATK+20%  (matk * 0.20)
MDEF+20%  (mdef * 0.20)
SPD +20%  (spd * 0.20)
```
灵宠无 MP 字段 → 光环不含 MP。`computePetAura(pet)`（src/systems/PetSystem.ts）统一计算，E2E 已校验公式对齐。

### 6.2 面板（U 键，`src/ui/panels.ts`）
- 顶部容器 + 遮罩 + `pauseForMenu`，`pointerdown` 守卫冻结移动（与 B/C/J 面板同范式）。
- 每张灵宠卡：图标方块(物种色)、名称、Lv、EXP 进度条、六维、技能名(若有)、出战徽章。
- 按钮：`出战/收回` 切换（toggle active）、`放生`（release）。
- 底部：出战灵宠光环预览行。
- C 面板新增「灵宠」紫色入口 → 关闭 C 打开灵宠面板。
- GameScene：U 键 `togglePetPanel()`；`refreshOpenPanels` 含灵宠面板即时刷新（worldSync 到达后重渲染）。

### 6.3 Dev 快捷键
`Ctrl+Y`：联机下 `requestPetGrantDev()` 发放随机物种灵宠（落库避免重连丢失）；离线提示需联机。

## 七、端到端烟测（`scripts/test-pet.ts`，E2E 28/28）

与真实联机完全一致：编译服务端 → 临时库起 Colyseus 权威服（随机端口）→ colyseus.js 客户端走生产路径。覆盖：

- **A 发放**：随机物种成功；首只自动出战（`pets.length=1, active=true`）
- **B 发放指定物种**：`fox_fire` 成功；长度=2 且仅一只出战
- **C 切换出战**：`setActive` 后仅目标出战，原出战收回
- **D 收回**：`recall` 后无出战灵宠
- **E 放生**：`release` 后长度递减；若放生的是出战则首只接替
- **F 栏位上限**：满 6 后第 7 只被拒（含「上限」提示）
- **G 持久化**：`world_data` 落库 pets=6、仅一只是出战；断连新客户端重连恢复=6
- **H 光环公式**：`computePetAura` 的 HP/ATK/DEF/MATK/MDEF/SPD 与 10%/20% 比例逐项对齐

运行：`npm run test:pet:e2e`（内部 `npx tsx scripts/test-pet.ts`，不污染真实 data.db）。

## 八、v1 范围外（延期项）

- **战斗协同（v2）**：灵宠在回合战斗中的行动/技能（赤焰狐灼烧、玄岩龟护盾、幼麟祥瑞光环等已埋 `skill` 字段，待战斗系统接入）。
- **获取途径**：当前仅 Dev 快捷键（`Ctrl+Y`）发放；正式应来自击败妖兽收服 / 特定区域捕捉 / 活动，留待后续接采集或战斗掉落。
- **忠诚度系统**：`loyalty` 字段已存但暂不影响数值（后续可做亲密度→光环加成或战斗触发率）。
- **灵宠升级经验来源**：`addPetExp` 已实现，但 v1 无客户端意图调用（经验来源待与战斗/任务挂钩）。
