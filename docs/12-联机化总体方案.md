# 12-联机化总体方案（大流程图）

> 版本：v0.1 | 创建：2026-07-09 | 作者：隐世 + 开发代理
> 适用范围：《解》v0.3.0 现状 → 联机优先路线图
> 原则：不炫技、不堆架构，solo 项目复杂度成正比；先用本地切片排雷，不一步到位。

---

## 一、核心决策（已拍板）

| 项 | 决策 |
|---|---|
| 联机定位 | 推翻原"首发决策不设多人组队"，改为 **联机优先** |
| 技术栈 | **Colyseus + Node + TS**（房间模型天然适配组队/副本/战斗实例，自带状态同步 + 重连） |
| 首切片范围 | 不止移动同步，**直接带权威战斗（组队打怪）** |
| 数据库 | 首切片不需要；到元系统阶段（Stage D）再引入 |
| 云服务器 | 开发期不需要；上线（Stage E）才考虑 |

---

## 二、架构总览（四层）

游戏联机 ≠ 普通 web 前后端分离。正确分层：

| 层 | 职责 | 技术形态 |
|---|---|---|
| **客户端** | 输入采集 + 渲染 + 动画表现 | Phaser（只当渲染器，不再当逻辑老大） |
| **权威游戏服** | 跑模拟：移动、战斗结算、伤害、掉落、状态 | Node 常驻进程 + WebSocket（Colyseus） |
| **服务后端** | 非实时元系统：账号、公会 CRUD、拍卖挂单 | REST/HTTP（Express 等） |
| **数据库** | 账号、角色、背包、公会持久化 | SQLite（本地）→ PostgreSQL（部署） |

> 注：REST 那套"前后端分离"只负责**账号/公会/拍卖**等非实时部分；**实时游戏逻辑走权威游戏服的 WebSocket，不是 REST**。

## 三、准备阶段（当前，轻量）

### 环境
- Node ≥ 16（已在跑 vite，必满足；Colyseus 0.15+ 要求 Node 16+）。

### 依赖（加到现有 `game/package.json`，不动客户端逻辑）
- `colyseus` — 服务端核心
- `colyseus.js` — 客户端连接 SDK
- `@colyseus/schema` — 状态同步（客户端服务端共用）
- `express` — 可选，发布时顺带托管客户端 build
- `tsx` — 直接跑 TS 服务端（免编译）
- `concurrently` — 一条命令同跑前后端
- `@esotericsoftware/spine-phaser` — 客户端 Spine 骨骼动画运行时（美术阶段接入，C 前；服务端不需）

### 脚手架结构（新增，不碰现有游戏代码）
```
game/
  server/                ← 新增
    index.ts             ← Colyseus 监听 2567 + 房间注册
    rooms/GameRoom.ts     ← 共享地图：移动同步 + 聊天
    rooms/BattleRoom.ts   ← 权威战斗：组队打怪
  shared/                ← 可选：schema 定义 + 意图(intent)类型（两端共用）
  tsconfig.server.json   ← 新增：module:NodeNext + types:["node"]
```

### npm scripts（补三条）
- `dev:client` → `vite`
- `dev:server` → `tsx watch server/index.ts`
- `dev` → `concurrently "npm:dev:client" "npm:dev:server"`

### 铁律
- **客户端构建图绝不 import 服务端入口**，否则 vite 打包崩。共享逻辑走 `src/systems` / `shared`，两端都引它。
- 现有 `tsconfig.json` 是 Vite 客户端用的（`ESNext` + `bundler`），**不能拿它编服务端**；服务端用独立的 `tsconfig.server.json`。

### 代码现状（利好）
- `src/systems/` 19 个文件（BattleData / BossMechanics / SkillMechanics / StatusSystem / GameState* / Zones / Inventory / Kido / QuestData / SaveManager / ConsumableSystem / EnhanceSystem / BestiaryData / Skills）**已完全不依赖 Phaser**（grep 零命中）→ 纯逻辑已解耦，服务端可直接 import，无需大改。

## 四、实施阶段（Stages）

### Stage A — 逻辑解耦与共享（大部分已完成）
- **现状**：`src/systems/` 已无 Phaser 依赖，纯逻辑解耦。
- **动作**：确认 systems 能在服务端 import 并调用；在 `shared/` 定义意图类型（如 `UseSkill`、`Move`、`Chat`）与状态 Schema。
- **验收**：服务端进程能 import `BattleData` / `SkillMechanics` / `StatusSystem` / `BossMechanics` 并跑出伤害结果。

### Stage B — 最小联机切片（共享地图房间）
- **GameRoom**：多客户端同房间、玩家位置同步、聊天。
- **验收**：两个浏览器标签进同一房间，互相看到移动 + 聊天。
- **美术**：可暂用英雄 sprite 染色代表其他玩家，不卡美术。

### Stage C — 权威战斗（组队打怪）
- **BattleRoom**：服务端跑回合循环与伤害结算；客户端发意图、播动画。
- **动作**：重构 `BattleScene` 从"我自己跑回合"改为"渲染房间状态 + 发意图"；**动画改为事件驱动**（收到服务端消息即 play 动画）。
- **验收**：组队进入战斗，所有客户端看到一致状态与动画，伤害由服务端算（反作弊天然解决）。

### Stage D — 元系统（需 DB）
- 账号 / 鉴权 + 数据库（SQLite 本地 → PostgreSQL 部署）。
- 组队管理 UI、公会、拍卖行（即用户原定的"框架类"系统）。
- **验收**：注册登录、角色持久化、公会 / 拍卖可用。

### Stage E（可选）— 部署上线
- 云主机 / 支持 WebSocket 长连接的 PaaS（Colyseus 有部署指引）。
- 反作弊、重连、扩容考量。

## 五、美术资源 / 骨骼动画（重点）

> **美术工具：Spine（Esoteric Software，一次性买断授权，工业级 2D 骨骼动画编辑器）。Phaser 3 通过官方插件 `@esotericsoftware/spine-phaser` 原生支持 Spine 运行时，导出物（`.json` 骨骼 + `.atlas` 图集 + `.png` 贴图）可直接加载播放。** 骨骼方案 vs 帧动画：同一套贴图，骨骼可程序化补间、省美术量、且换肤（skin）机制天然适配套装 / 坐骑外观变更（只换贴图不换骨架）。

**结论：美术与 netcode 正交，不阻塞架构。** 动画做成"事件驱动"最省事，多人下远端玩家动画天然同步，以后不返工。

| 阶段 | 是否需要新美术 | 具体美术 · 动画需求 |
|---|---|---|
| Prep / A | 否 | — |
| B 共享地图 | 部分 | 其他玩家角色 Spine 骨架 + 行走/待机动画（切片可染色英雄顶替）；nameplate（头顶名） |
| C 权威战斗 | 部分 | 其他玩家战斗立绘 + 技能动画（Spine 触发，复用现有骨架/贴图）；伤害飘字同步 |
| D 元系统 | UI 为主 | 公会 / 拍卖 / 组队面板（轻量 2D UI 资源） |
| 内容更新（套装 / 坐骑 / 灵宠 / 新区域） | 是 | Spine 骨架 + 专属贴图：坐骑行走/骑乘、灵宠待机/攻击、套装换肤、新敌骨架 |

### Spine 生产 → 引擎（独立工作流，随时可做）
1. **美术产出**：Spine 编辑器绑定骨架、K 帧、导出 `.json`（骨骼）+ `.atlas`（图集）+ `.png`（贴图）。
2. **引擎加载**：`spine-phaser` 加载导出物 → `spine.SpineGameObject`（`this.add.spine(...)`）。
3. **触发播放**：本地输入 **或** 服务端事件（`room.onMessage` → `spineObj.animationState.setAnimation(0, name, loop)`）——同一套播放接口。
4. **多人同步要点**：动画由"收到服务端事件"触发，不绑死本地输入；远端玩家动画天然同步，无需额外同步动画状态。
5. **换肤（skin）**：套装 / 坐骑通过 Spine skin 切换贴图，零改骨架——内容更新期复用同一骨架只换 skin。

### 建议时间线
- **现在（Prep ~ C）**：用现有 / 复用 sprite，不卡美术；先接入 `spine-phaser` 运行时验证。
- **C 完成时**：把战斗 / 移动动画统一改为事件驱动（趁重构 BattleScene 一次到位）。
- **内容更新期**：在 Spine 里产出套装 / 坐骑 / 灵宠 / 新敌骨架与 skin。

## 六、风险与注意

- **复杂度乘数**：全联机 = auth / DB / 托管 / 反作弊 / 重连 / 扩容，每项独立大坑。solo 先 B 切片排雷。
- **tsconfig 隔离**：客户端 `ESNext`+`bundler` vs 服务端 `NodeNext`+`node` 类型，不能混。
- **客户端构建绝不 import 服务端入口**。
- **WebSocket 托管坑**：选支持长连接的平台（部分 PaaS 不支持）。
- **不要现在做**：装 Postgres/MySQL、租云服务器、建账号鉴权、碰公会/拍卖、上 Docker/K8s。
